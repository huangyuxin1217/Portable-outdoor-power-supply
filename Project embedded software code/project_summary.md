# ESP32 控制数学系统 - 工程总结

本文档总结了当前 ESP32 控制系统的工程架构、运行流程及各核心模块的优化策略。

## 1. 工程架构 (Architecture)

本工程基于 **ESP32** 平台，采用 **FreeRTOS** 实时操作系统架构，实现了多任务并发处理。

### 核心硬件抽象层
*   **主控**: ESP32-WROOM
*   **协处理器通信**: 通过 UART2 与 STM32F103C8T6 通信 (数据采集)
*   **交互**:
    *   **输出**: 0.96寸 OLED (SSD1306), WS2812B 彩灯条, MAX98357A I2S 音频放大器
    *   **输入**: I2S 全向麦克风 (INMP441/SPH0645)

### 软件任务架构
系统通过 FreeRTOS 任务和 Arduino `loop()` 结合的方式运行：

1.  **Main Loop (Core 1)**: 负责高优先级的非阻塞逻辑
    *   Web Server 请求处理
    *   MP3 音频流播放 (`mp3->loop()`)
    *   LED 动画效果计算与刷新
    *   OLED 界面刷新 (500ms 间隔)
    *   语音指令响应逻辑
2.  **Voice Task (Core 1, Pinned)**: 语音识别专用任务
    *   持续读取 I2S 音频流
    *   执行 Edge Impulse 神经网络推理
    *   通过队列 (`voiceQueue`) 发送识别结果
3.  **STM32 Comm Task (Core 1, Pinned)**: 串口通信任务
    *   监听 UART2 接收缓冲区
    *   负责数据帧解析、校验和验证
    *   更新共享数据结构 (互斥锁保护)
4.  **WiFi Reconnect Task (Core 1, Pinned)**: 后台网络维护
    *   监控 WiFi 连接状态
    *   断线自动重连，确保主流程不卡顿

## 2. 程序运行流程 (Execution Flow)

### 2.1 启动流程
1.  **硬件初始化**: 初始化串口、LED、OLED、I2S 总线、SPIFFS 文件系统。
2.  **网络启动**: 尝试连接 WiFi (非阻塞)，成功则启动 Web Server 并播放欢迎音，失败则通过后台任务重试。
3.  **任务创建**: 启动语音识别任务、STM32 通信接收任务、WiFi 守护任务。
4.  **进入主循环**: 开始轮询处理各类事件。

### 2.2 语音控制交互流程
1.  **唤醒**: 用户说出 "Hello" -> 识别成功 -> 播放 "yes.mp3" -> 进入**激活状态** (8秒超时)。
2.  **指令**:
    *   用户说 "KaiDeng" (仅在激活状态有效) -> 打开 LED (黄色) -> 播放 "on.mp3"。
    *   用户说 "GuanDeng" (仅在激活状态有效) -> 关闭 LED -> 播放 "off.mp3"。
3.  **反馈**: 每次识别成功都会通过串口打印置信度，并在 LED 上显示简单的反馈动画（如唤醒时亮蓝灯）。

### 2.3 数据监测流程
1.  **STM32 发送**: 打包电压、电流、温度等传感器数据 -> 添加帧头/帧尾/校验 -> UART 发送。
2.  **ESP32 接收**: `stm32Task` 捕获数据 -> 校验通过 -> 更新全局 `STM32Data` 结构体。
3.  **显示更新**: 主循环每 500ms 读取一次最新数据 -> 刷新 OLED 显示屏 -> 供 Web 端 API 查询。

## 3. 各模块优化 (Optimizations)

### 3.1 语音识别 (Voice Recognition)
*   **滑动窗口机制 (Sliding Window)**: 每次推理仅更新 1/3 的音频数据，保留 2/3 旧数据。这极大提高了对跨窗口关键词的捕捉能力，避免"掐头去尾"导致的识别失败。
*   **数字增益 (Digital Gain)**: 在读取 I2S 数据时通过位移操作 (`raw >> 14`) 调整输入音量，适配不同麦克风灵敏度。
*   **激活词逻辑 (Wake Word)**: 引入 "Hello" 作为激活词，只有唤醒后才响应控制指令，有效防止日常谈话中的误触发。
*   **冷却防抖**: 识别成功后有一秒钟的“不应期”，防止一次发音被连续识别多次。

### 3.2 通信模块 (Communication)
*   **稳健协议**: 设计了包含 `帧头(AA 55)`、`长度`、`校验和`、`帧尾(0D 0A)` 的数据包格式，极大降低了串口通信的误码率和错包率。
*   **线程安全**: 使用 FreeRTOS `Mutex` (互斥锁) 保护共享数据区的读写，防止多任务并发导致的数据竞争。
*   **状态机解析**: 接收逻辑采用状态机模式，能从字节流中精准提取数据帧，具备自动丢弃乱码和错误恢复能力。

### 3.3 系统体验 (User Experience)
*   **非阻塞 WiFi**: WiFi 连接失败不会阻塞系统启动，允许设备在离线状态下正常使用语音和本地控制功能。
*   **Web 控制台**: 提供直观的网页界面 (Vue-like 风格)，支持实时数据监视、颜色拾取器、LED 动效切换和远程消息发送。
*   **OLED 动态显示**: 当收到 Web 端消息时，OLED 会暂时中断监测数据显示消息内容，10秒后自动恢复，提升交互性。

## 4. 总结
该工程成功整合了 **端侧 AI (语音)**、**实时控制 (STM32/LED)** 和 **IoT 网络 (Web)** 三大功能。代码结构清晰，通过任务分离保证了系统的实时性和稳定性，具备较高的实用价值和扩展性。
